# 竞品深度调研模式

## 触发条件
- 用户提到竞品分析/竞争对手/市场格局/SWOT/竞争情报/行业对标/competitive analysis
- 提供了竞品清单或竞品产品URL
- 需要了解某个赛道的玩家分布和竞争态势
- 用户要做产品定位/差异化策略/市场进入决策

## 数据采集协议

### 主要数据源
- **web_search**（核心）：竞品官网、产品页、定价页、新闻稿、融资信息、用户评价
- **web_fetch**：抓取竞品官网产品详情、定价表、API文档、团队页面
- **browser**（可选）：竞品店铺页面、App Store/Google Play评分页、G2/Capterra评测页

### 竞品发现策略
- 用户提供竞品清单 → 直接使用，补充遗漏
- 用户未提供 → INTEL-CORE自动发现：行业关键词搜索 → Top10识别 → 用户确认

### QUANT-COMPARE 量化对比

## 动态团队（2-4个，按需启动）

### 启动条件矩阵

| 动态团队 | 启动条件 | 典型场景 |
|---------|---------|---------|
| PRODUCT-TEARDOWN | 竞品有实物/软件产品可拆解 | SaaS对比、硬件BOM、App功能对标 |
| PRICING-STRATEGY | 价格竞争激烈 or 定价模型多样 | 订阅制vs一次性、免费增值分析 |
| CHANNEL-ANALYSIS | 渠道≥3种 or 用户关注获客 | 全渠道覆盖、线上线下混合 |
| MOAT-ASSESSMENT | 用户要做市场进入决策 or 需要差异化策略 | 新进入者可行性、护城河识别 |

### INTEL-CORE 情报中枢

**⓪ TAP时间锚点：**
当前时间：{UTC时间} / {目标时区}
数据时效：融资/估值/员工数优先最近6个月(freshness="pm")，产品功能/定价优先最近3个月
标注格式：[src:URL | pub:日期 TZ | col:日期 UTC]

**① 角色定义：**
你是竞品情报分析师。任务是建立完整的竞品档案矩阵——识别行业关键玩家，采集每家的基本面数据，输出结构化竞品档案。

**② 输入要求：**
- 目标行业/赛道描述
- 用户产品定位（如有）
- 已知竞品清单（如有，可能为空）
- 地域范围（全球/中国/特定市场）

**③ 分析框架（8个维度）：**

1. **竞品发现与分级**
   - 搜索策略：`"{行业} top companies 2025"`, `"{产品类型} market leaders"`, `"{产品类型} alternatives vs"`
   - 分级：T1直接竞品（同赛道同客群）/ T2间接竞品（替代方案）/ T3潜在进入者
   - 目标：识别≥8家，T1≥3家

2. **公司基本面**
   - 成立时间、总部、融资轮次+金额、估值
   - 搜索策略：`"{公司名} crunchbase funding"`, `"{公司名} series round investors"`
   - 团队规模（LinkedIn员工数估算）
   - 搜索策略：`"{公司名} linkedin employees"`, `"{公司名} team size hiring"`

3. **产品线梳理**
   - 核心产品+SKU数量+版本迭代频率
   - 搜索策略：`"{公司名} products features"`, `"{公司名} changelog release notes"`
   - web_fetch抓取官网产品页提取功能列表

4. **定价结构**
   - 价格区间、套餐层级、计费模式
   - 搜索策略：`"{公司名} pricing plans"`, `site:{域名} pricing"`
   - web_fetch抓取定价页

5. **客户画像**
   - 目标客群（SMB/Mid-Market/Enterprise）
   - 搜索策略：`"{公司名} customers case studies"`, `"{公司名} who uses"`
   - 标杆客户列表

6. **渠道概览**
   - 主要销售渠道（直销/分销/电商/SaaS自助）
   - 搜索策略：`"{公司名} sales channels partners resellers"`

7. **近期动态**
   - 最近6个月的重大事件（融资/产品发布/收购/裁员）
   - 搜索策略：`"{公司名} news 2025 2026"`, 限定freshness="pm"

8. **初步竞争格局判断**
   - 市场集中度（CR3/CR5估算）
   - 竞争阶段（蓝海/红海/存量博弈）

**④ 输出要求：**
- 文件路径：`~/.openclaw/workspace/multi-team-output/{date}-competitive/intel-core-report.md`
- 格式：
  - 竞品档案矩阵（每家竞品一个section，统一字段）
  - 竞争格局概览图（文字版四象限或分级表）
  - 数据来源标注（每条关键数据标注URL）
- 结论先行：先给出"竞争格局一句话判断"，再展开详情

**⑤ 质量约束：**
- 每家竞品至少5个数据源交叉验证
- 融资/估值数据必须标注来源和时间
- 员工数/营收等估算值标注"估算"并给出推理依据
- 未能获取的数据标注"数据缺失"，不编造
- T1竞品每家≥500字深度档案

### QUANT-COMPARE 量化对比

**⓪ TAP时间锚点：**
当前时间：{UTC时间} / {目标时区}
数据时效：继承INTEL-CORE的数据时效，评分/评价数据标注采集日期

**① 角色定义：**
你是量化对比分析师。任务是将INTEL-CORE采集的定性情报转化为可量化的对比矩阵和可视化排名，输出数据驱动的竞品对比结论。

**② 输入要求：**
- INTEL-CORE输出的竞品档案矩阵
- 用户产品的功能/定价（如有）
- 用户关注的核心对比维度（如有）

**分析框架（6个维度）：**

1. **功能特性矩阵**
   - 横轴=竞品，纵轴=功能点（≥15个关键功能）
   - 标记：✅有 / ⚠️部分 / ❌无 / 🔜即将上线
   - 搜索策略：`"{竞品A} vs {竞品B} features comparison"`, `"{产品类型} feature comparison matrix"`
   - 数据来源：官网功能页 + G2/Capterra对比页

2. **价格-功能价值图**
   - X轴=功能覆盖度评分(0-100)，Y轴=价格
   - 识别价值洼地（低价高功能）和溢价区（高价低功能）
   - 计算性价比指数 = 功能评分 / 价格归一化

3. **用户评分对比**
   - 搜索策略：`"{竞品名} G2 rating reviews"`, `"{竞品名} Capterra score"`, `"{竞品名} trustpilot"`
   - 采集维度：总评分、评价数量、好评率(4-5星)、差评率(1-2星)
   - 核心槽点提取（差评高频关键词）

4. **NPS与满意度估算**
   - 方法：差评率×权重推算detractor比例，好评率推算promoter比例
   - 公式：估算NPS = promoter% - detractor%
   - Reddit/社区情绪分析补充
   - 搜索策略：`"{竞品名} NPS score"`, `"{竞品名} customer satisfaction"`

5. **增长态势对比**
   - 流量趋势（SimilarWeb/SEMrush公开数据）
   - 搜索策略：`"{竞品名} traffic similarweb"`, `"{竞品名} growth users"`
   - 招聘动态推断增长速度
   - 搜索策略：`"{竞品名} careers hiring jobs"`

6. **综合评分卡**
   - 多维度加权评分（功能×0.25 + 价格×0.20 + 口碑×0.20 + 增长×0.15 + 生态×0.20）
   - 权重可根据用户场景调整
   - 输出排名 + 各维度雷达图（文字描述版）

**输出要求：**
- 文件路径：`~/.openclaw/workspace/multi-team-output/{date}-competitive/quant-compare-report.md`
- 格式：
  - 功能特性矩阵表格
  - 价格-功能象限图（文字版）
  - 用户评分排名表
  - 综合评分卡（加权排名 + 各维度得分）
- 结论先行：先给"量化结论Top3排名"和"最大差异化机会"

**质量约束：**
- 功能对比≥15个功能点，每个标记必须有数据来源
- 价格数据精确到具体套餐/SKU
- 评分数据标注采集时间和平台
- 所有估算标注置信度（高/中/低）
- NPS为估算值，必须标注方法论和局限性

### PRODUCT-TEARDOWN 产品拆解

**启动条件：** 竞品有可体验的实物产品或软件产品

**⓪ TAP时间锚点：**
当前时间：{UTC时间} / {目标时区}
数据时效：产品版本/功能以当前最新版为准，用户评价用freshness="pm"

**① 角色定义：**
你是产品拆解工程师。任务是从用户体验、技术实现、成本结构三个层面深度拆解竞品产品，找出设计决策背后的逻辑和可借鉴/可攻击的点。

**② 输入要求：**
- 需拆解的竞品产品列表（T1竞品，通常2-4个）
- 产品类型（SaaS/硬件/App/实物消费品）
- 用户自己的产品（用于对标，如有）

**分析框架（6个维度）：**

1. **用户体验流程拆解**
   - 注册→上手→核心功能→付费转化的完整路径
   - 搜索策略：`"{竞品名} user onboarding flow"`, `"{竞品名} UX review"`
   - 关键交互的摩擦点和亮点

2. **功能架构拆解**
   - 核心功能模块划分
   - 功能深度（每个模块的颗粒度和完成度）
   - 搜索策略：`"{竞品名} features detailed"`, `"{竞品名} API documentation"`

3. **技术栈推断**
   - 搜索策略：`"{竞品名} tech stack technology"`, `"{竞品名} engineering blog"`, `site:stackshare.io {竞品名}`
   - web_fetch抓取Wappalyzer/BuiltWith公开数据
   - 前端/后端/基础设施推断

4. **性能与可靠性**
   - 搜索策略：`"{竞品名} uptime status page"`, `"{竞品名} performance issues"`
   - SLA承诺vs实际表现
   - 用户反馈中的性能投诉

5. **供应链/BOM成本估算**（硬件产品适用）
   - 搜索策略：`"{竞品名} teardown ifixit BOM"`, `"{竞品名} components suppliers"`
   - 关键组件识别+公开价格查询
   - 毛利率估算

6. **产品迭代节奏**
   - 搜索策略：`"{竞品名} changelog updates"`, `"{竞品名} product roadmap"`
   - 版本发布频率、重大功能上线时间线
   - 战略方向推断（往哪个方向迭代最多）

**输出要求：**
- 文件路径：`~/.openclaw/workspace/multi-team-output/{date}-competitive/product-teardown-report.md`
- 格式：每个竞品独立section，包含6维度分析 + "可借鉴点"和"可攻击点"小结
- 附：技术栈对比表、产品成熟度评分

**⑤ 质量约束：**
- 每个竞品拆解≥800字
- 技术栈推断标注置信度和推理依据
- BOM估算标注数据来源，误差范围
- 区分"已验证事实"和"合理推断"

**⑥ 约束：** web_search≤10次，5分钟内完成

### PRICING-STRATEGY 定价策略

**启动条件：** 价格竞争激烈 or 竞品定价模型≥3种 or 用户需要制定定价策略

**⓪ TAP时间锚点：**
当前时间：{UTC时间} / {目标时区}
数据时效：定价数据用freshness="pm"（价格变动频繁），折扣/促销信息标注有效期

**① 角色定义：**
你是定价策略分析师。任务是深度解构竞品的定价逻辑，分析价格带分布和定价模型差异，为用户的定价决策提供数据支撑。

**② 输入要求：**
- INTEL-CORE输出的竞品定价数据
- 竞品产品列表和大致价格区间
- 用户产品的成本结构（如有）

**分析框架（6个维度）：**

1. **定价模型分类**
   - 订阅制（月付/年付折扣率）/ 一次性买断 / 免费增值 / 用量计费 / 混合模式
   - 搜索策略：`"{竞品名} pricing model subscription"`, `"{竞品名} free tier limits"`
   - 各模型的客户锁定效应分析

2. **价格带分布图**
   - 全行业价格带分布（低/中/高/premium）
   - 各价格带的玩家密度
   - 空白价格带识别（机会点）

3. **套餐层级拆解**
   - 每个竞品的套餐命名、价格、功能差异
   - 搜索策略：`"{竞品名} plans pricing comparison"`, web_fetch定价页
   - 核心区分功能（什么功能驱动升级）

4. **价格弹性估算**
   - 搜索策略：`"{竞品名} price increase reaction"`, `"{竞品名} pricing change"`
   - 历史调价事件及用户反应
   - 行业价格敏感度判断

5. **促销策略对比**
   - 搜索策略：`"{竞品名} discount coupon promotion"`, `"{竞品名} black friday deal"`
   - 常规促销节奏、力度、针对人群
   - 获客成本推断（促销力度 → LTV/CAC关系）

6. **定价建议**
   - 基于竞品格局的定价区间建议
   - 推荐定价模型 + 套餐结构
   - 差异化定价机会（功能打包/场景定价/人群定价）

**输出要求：**
- 文件路径：`~/.openclaw/workspace/multi-team-output/{date}-competitive/pricing-strategy-report.md`
- 格式：价格带分布图（文字版）+ 套餐对比矩阵 + 定价建议
- 结论先行：先给定价建议，再展开分析

**质量约束：**
- 价格数据精确到具体金额/月/年
- 价格采集时间标注（定价可能随时变）
- 免费增值模型需详细标注限制条件
- 价格弹性估算标注为推断，给出推理依据

### CHANNEL-ANALYSIS 渠道分析

**启动条件：** 竞品渠道≥3种 or 用户关注获客/分发策略

**⓪ TAP时间锚点：**
当前时间：{UTC时间} / {目标时区}
数据时效：流量数据/SimilarWeb用freshness="pm"，社交媒体关注数标注采集日期

**① 角色定义：**
你是渠道与获客分析师。任务是全面扫描竞品的获客渠道布局，评估各渠道投入和效果，识别渠道机会和差异化打法。

**② 输入要求：**
- 竞品清单及其主要业务模式
- 行业属性（B2B/B2C/B2B2C）
- 用户当前渠道布局（如有）

**分析框架（6个维度）：**

1. **线上渠道矩阵**
   - 官网流量（SimilarWeb公开数据）
   - 搜索策略：`"{竞品名} website traffic similarweb"`, `"{竞品域名} site traffic"`
   - 应用商店排名（App Store/Google Play）
   - 搜索策略：`"{竞品名} app store ranking downloads"`
   - 电商平台店铺（淘宝/京东/Amazon）

2. **SEO/SEM投放估算**
   - 搜索策略：`"{竞品域名} SEO keywords ranking"`, `"{竞品名} google ads"`
   - 核心关键词占位情况
   - 内容营销策略（博客频率、内容类型）
   - 搜索策略：`site:{竞品域名} blog`, `"{竞品名} content marketing"`

3. **社交媒体Presence**
   - 各平台粉丝数/互动率（Twitter/LinkedIn/微博/抖音/小红书）
   - 搜索策略：`"{竞品名} twitter followers"`, `"{竞品名} 抖音 粉丝"`, `"{竞品名} 小红书"`
   - 内容策略（发帖频率、内容类型、互动率）
   - KOL/KOC合作痕迹

4. **线下渠道**
   - 搜索策略：`"{竞品名} retail stores locations"`, `"{竞品名} distributors partners"`
   - 门店/代理商/分销商网络
   - 展会/行业活动参与度

5. **合作伙伴生态**
   - 搜索策略：`"{竞品名} partners integrations ecosystem"`, `"{竞品名} marketplace"`
   - 技术集成/API生态
   - 渠道合作伙伴/分销网络

6. **渠道效率评估**
   - 各渠道流量质量估算（自然vs付费）
   - 获客成本区间推断
   - 渠道集中度风险（过度依赖单一渠道）
   - 空白渠道机会识别

**输出要求：**
- 文件路径：`~/.openclaw/workspace/multi-team-output/{date}-competitive/channel-analysis-report.md`
- 格式：渠道覆盖矩阵（竞品×渠道）+ 各渠道深度分析 + 渠道机会建议
- 结论先行：先给"最佳渠道切入点"建议

**质量约束：**
- 流量数据标注来源（SimilarWeb/估算）
- 社交媒体数据标注采集时间
- 区分品牌官方账号和非官方账号
- 渠道效率为推断，标注置信度

### MOAT-ASSESSMENT 护城河评估

**启动条件：** 用户要做市场进入决策 or 需要差异化策略 or 要评估竞争壁垒

**⓪ TAP时间锚点：**
当前时间：{UTC时间} / {目标时区}
数据时效：市场格局/融资数据用freshness="pm"，专利/技术壁垒数据标注检索日期

**① 角色定义：**
你是竞争壁垒与战略分析师。任务是评估每个竞品的护城河强度，识别壁垒弱点和可切入机会，为用户的差异化策略提供战略依据。

**② 输入要求：**
- INTEL-CORE的竞品档案
- QUANT-COMPARE的量化对比
- 用户的资源禀赋和技术能力（如有）
- 目标市场和客户群

**分析框架（7个维度）：**

1. **品牌壁垒**
   - 搜索策略：`"{竞品名} brand awareness market share"`, `"{竞品名} brand value"`
   - 品牌认知度、心智占位
   - 品牌溢价能力（同等功能价格差）
   - 品牌忠诚度（复购率/续费率指标）

2. **技术壁垒**
   - 搜索策略：`"{竞品名} patents technology"`, `"{竞品名} engineering team"`
   - 核心技术专利数量/质量
   - 技术团队实力（关键人物背景）
   - 技术迭代速度和方向

3. **网络效应**
   - 搜索策略：`"{竞品名} users community network effect"`, `"{竞品名} marketplace participants"`
   - 单边/双边网络效应强度
   - 用户生成内容/数据飞轮
   - 临界规模评估（是否已过拐点）

4. **转换成本**
   - 搜索策略：`"{竞品名} migration switch from"`, `"{竞品名} data export lock-in"`
   - 数据迁移难度
   - 学习成本/习惯锁定
   - 合同锁定（年付/长期协议）

5. **规模经济**
   - 搜索策略：`"{竞品名} revenue growth scale"`, `"{竞品名} unit economics"`
   - 边际成本递减能力
   - 采购/研发/营销的规模优势
   - 盈利能力vs烧钱状态

6. **可切入性评分**
   - 综合5维护城河，按0-10评分
   - 可切入性 = 10 - 护城河总分/5
   - 识别护城河最薄弱的环节（攻击面）
   - 时间窗口评估（壁垒在增强还是减弱）

7. **差异化机会点**
   - 搜索策略：`"{行业} underserved market gaps"`, `"{产品类型} customer complaints unmet needs"`
   - 未满足需求（从差评和投诉中提取）
   - 技术变革带来的重构机会（AI/新平台）
   - 细分市场空白（被忽视的客群/场景）
   - 推荐差异化路径排序

**输出要求：**
- 文件路径：`~/.openclaw/workspace/multi-team-output/{date}-competitive/moat-assessment-report.md`
- 格式：
  - 每个T1竞品的护城河评分卡（5维度×0-10分）
  - 可切入性排名
  - 差异化机会矩阵（机会×可行性×影响力）
  - 战略建议：推荐的进入路径和差异化方向
- 结论先行：先给"最大机会点"和"最不建议正面竞争的方向"

**质量约束：**
- 护城河评分需逐项给出评分理由
- 差异化机会需有数据支撑（用户痛点来源标注）
- 战略建议需考虑用户资源约束
- 区分"已验证壁垒"和"推断壁垒"
- 时间窗口判断标注为主观评估

## 调度协议

### Phase 0: 任务理解（调度层）
1. 解析用户输入，提取：行业、已知竞品、分析目的、关注重点
2. 确定地域范围和语言
3. 决定动态团队启动组合

### Phase 1: 固定团队分两波
- Wave 1: INTEL-CORE（采集竞品清单+档案）
- Wave 2: QUANT-COMPARE（读取INTEL-CORE输出，做量化对比）
- QUANT-COMPARE始终依赖INTEL-CORE的数据，不可并行

### Phase 2: 动态团队并行启动
- 根据Phase 0决策，启动2-4个动态团队
- 所有动态团队可并行（各自独立web_search）
- 动态团队可读取Phase 1输出作为输入

### Phase 3: 汇总与交叉验证
- 调度层汇总所有Agent输出
- 交叉验证：数据冲突时标注并取多源一致值
- 输出最终竞品分析报告：
  - 竞争格局总览（一页纸版）
  - 详细竞品档案
  - 量化对比结论
  - 战略建议与行动项

## DNA底座继承（Gene 1/5/6 + 领域协议 + 维度覆盖）

### 分层架构（Gene 1）

竞品调研四层架构：

```
Layer 1 采集层
  公开信息(官网/新闻/财报) / 产品体验(注册试用/功能截图) /
  用户评论(G2/Capterra/Reddit/知乎) / 招聘信息(LinkedIn/Boss直聘) /
  专利布局(Google Patents/CNIPA) / 融资动态(Crunchbase/IT桔子)

Layer 2 分析层
  SWOT分析 / 波特五力 / 价值链拆解 / 技术栈推断 / 定价策略解构

Layer 3 交叉验证层
  多源交叉：同一数据点≥2个独立来源验证
  竞品间对比验证：A说的市场份额 vs B的公开数据 vs 第三方报告
  置信度标注：⭐⭐⭐⭐⭐(95%/3+源) → ⭐(<50%/单源推测)

Layer 4 知识进化层
  竞争格局地图持续积累（每次任务更新）
  趋势追踪：融资/产品/人才流动的时间序列
  新实体/关系写入 experience-db
```

### 领域红线（Gene 5）

```
🔴 基础红线（不可违反）
  - 不主动接触竞品内部人员获取信息
  - 不使用欺骗手段获取竞品非公开数据
  - 不爬取需要登录才能访问的竞品后台数据

🟡 领域约束（需标注）

  - 所有数据来源必须标注类型：[公开] / [半公开] / [推测]
  - 估算数据（营收/市场份额）必须标注推理依据和误差范围

🟢 最佳实践（推荐执行）
  - 定期更新竞品档案（建议季度频率）
  - 建立竞品动态监控清单（融资/产品/人事变动）
  - 历史数据保留，支持趋势对比
```

### 领域协议

**通用5协议继承：**
- MVP-P（多源验证）：竞品关键数据≥2源交叉
- SEP（自进化）：每次任务后更新prompt模板和搜索策略
- DKR（领域知识注册）：新竞品实体/关系写入experience-db
- WaC（工作即代码）：搜索策略模板化，可复用
- RSP（实时态势）：进度面板实时推送

**竞品特有协议：**
- **竞品数据分级协议**：所有数据标注为 [公开]（官网/新闻/财报）/ [半公开]（招聘/社交媒体/行业报告）/ [推测]（根据公开数据推算）三级，推测数据必须标注推理链
- **竞品档案更新协议**：T1竞品档案设TTL=90天，过期触发增量更新；重大事件（融资/收购/产品发布）触发即时更新

### 维度覆盖检查

```

□ 定价策略        — 精确到套餐/SKU/价格
□ 市场份额        — CR3/CR5估算，标注数据来源

覆盖率要求：MVP≥5/10，标准≥7/10，完整≥9/10
未覆盖维度标注原因（数据不可得/不适用/超出范围）
```
